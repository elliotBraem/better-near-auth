# Better-Near-Auth Implementation Guide

## Overview

Better-near-auth is a Better Auth plugin implementing Sign in with NEAR (SIWN, NEP-413) plus a built-in NEP-366 delegate action relayer. It provides:

- **SIWN authentication** — wallet-based sign-in with automatic single-step/two-step flow detection
- **Gasless relay** — server-side relayer broadcasts delegate actions on behalf of authenticated users, paying gas from a relayer account
- **Ephemeral relayer keypair** — auto-generated ED25519 keypair on startup, private key encrypted with AES-256-GCM in the database, survives server restarts
- **Profile integration** — FastNear KV primary, NEAR Social fallback

## Architecture

```
┌─ Web App ─────────────────────────────────────────┐
│  siwnClient({ recipient, networkId })              │
│  authClient.near.buildSignedDelegateAction(...)    │
│  authClient.near.relayTransaction(...)             │
└──────────────────┬────────────────────────────────┘
                   │ signed delegate action (FAK)
                   ▼
┌─ Better Auth Server ──────────────────────────────┐
│  /near/relay — validates session + whitelist,     │
│                 wraps in tx, signs with relayer    │
│                 key, broadcasts via FastNear RPC    │
│  /near/relay-status/:txHash                        │
│  /near/relayer-info                                │
│  /near/nonce, /near/verify, /near/profile, etc.   │
└──────────────────┬─────────────────────────────────┘
                   │ broadcast_tx_commit
                   ▼
              NEAR Protocol (on-chain)
```

```typescript
siwn({
  // SIWN options

  validateRecipient?: (recipient: string) => boolean;
  validateMessage?: (message: string) => boolean;
  getProfile?: (accountId: string) => Promise<Profile | null>;
  validateLimitedAccessKey?: (args: { accountId, publicKey, recipient? }) => Promise<boolean>;

  // FastNear
  fastnearApiKey?: string;              // or process.env.FASTNEAR_API_KEY

  // Relayer
  relayer?: {
    accountId?: string;                  // explicit mode (production)
    privateKey?: string;                 // explicit mode (base64)
    relayTarget?: string;                // RPC URL (default: FastNear RPC)
    whitelistedContracts?: string[];     // restrict relay targets
    maxGasPerTransaction?: string;       // default "300 Tgas"
    maxDepositPerTransaction?: string;   // default "0"
  };
  // When accountId + privateKey omitted → ephemeral mode
})
```

## Relayer Modes

### Ephemeral (default)

When `relayer` is provided without `accountId`/`privateKey`:

1. On startup, checks DB for existing `relayerKey` record
2. If found: decrypts private key using `BETTER_AUTH_SECRET` as KEK (HKDF-SHA256 → AES-256-GCM), recovers same accountId
3. If not found: generates ED25519 keypair, derives implicit accountId (hex of public key), encrypts + stores in DB, logs accountId
4. Private key lives only in process memory — never in env vars or config files

Startup log:
```
[siwn] Relayer created in EPHEMERAL mode: 7a3c4b5c... (mainnet)
[siwn] Fund this account with NEAR to enable gasless relay
[siwn] Private key is encrypted in DB — persists across restarts
```

### Explicit (production)

When `accountId` + `privateKey` are provided in config — for named, persistent relayer accounts like `relayer.myapp.near`.

## Delegate Action Flow (NEP-366)

```
1. User signs in via SIWN → gets function-call access key (FAK)
2. FAK is scoped to the recipient contract
3. Client calls buildSignedDelegateAction({ receiverId, actions })
   - Queries access key nonce + block height via FastNear RPC
   - Builds DelegateAction struct
   - Signs via wallet's signDelegateActions (internal API)
   - Returns base64 borsh-encoded SignedDelegateAction
4. Client calls relayTransaction({ signedDelegateAction })
   - Server validates: session authenticated, sender matches, receiver whitelisted
   - Server wraps delegate action in Transaction signed with relayer key
   - Broadcasts via FastNear RPC broadcast_tx_commit
5. Transaction executes on-chain as if the user sent it
   - Relayer pays gas
   - predecessor_id = user's accountId
```

## Profile Lookup

Profile lookup tries FastNear KV first, falls back to NEAR Social:

1. `GET https://kv.main.fastnear.com/v0/latest/social.near/{accountId}/profile/**`
2. Fallback: `POST https://api.near.social/get` with `{ keys: ["{accountId}/profile/**"] }`

## Encryption Details

- **KEK derivation**: HKDF-SHA256 with `BETTER_AUTH_SECRET`, salt `better-near-auth-relayer`, info empty
- **Encryption**: AES-256-GCM with 12-byte random IV
- **Storage**: `encryptedPrivateKey` (base64 of ciphertext) + `iv` (base64 of IV) in `relayerKey` table
- **Trust model**: Same as Better Auth session tokens — DB access + `BETTER_AUTH_SECRET` = full access

## signDelegateActions Note

`@fastnear/wallet` (v0.10.4) implements `signDelegateActions` on all internal wallet adapter types (ParentFrameWallet, SandboxWallet, InjectedWallet) but does NOT export it from the public API surface. The client accesses it via the internal `connectedWallet` object. A public export has been requested — the code will switch to the public API once available.

## Wallet Connection

Uses `@fastnear/wallet` public API:

- `nearWallet.connect({ network, contractId })` — opens wallet picker
- `nearWallet.disconnect()` — disconnects
- `nearWallet.signMessage({ message, recipient, nonce })` — NEP-413 signing
- `nearWallet.accountId()` — current account
- `nearWallet.availableWallets()` — list manifests
- `nearWallet.selectWallet()` — show selector, returns wallet ID
- `nearWallet.restore({ network, contractId })` — rehydrate on page load
- `nearWallet.onConnect(cb)` / `nearWallet.onDisconnect(cb)` — event listeners

## RPC Calls

All on-chain queries and transaction broadcasting use FastNear RPC endpoints:

- Mainnet: `https://rpc.mainnet.fastnear.com`
- Testnet: `https://rpc.testnet.fastnear.com`

API key passed as `?apiKey=` query parameter when configured.

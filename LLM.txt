# Better-Near-Auth Implementation Guide

## Overview

Better-near-auth is a Better Auth plugin implementing Sign in with NEAR (SIWN, NEP-413) plus a built-in NEP-366 delegate action relayer. It provides:

- **SIWN authentication** — wallet-based sign-in with single-step/two-step flow
- **Gasless relay** — server-side relayer broadcasts delegate actions on behalf of authenticated users, paying gas from a relayer account
- **Ephemeral relayer keypair** — auto-generated ED25519 keypair on startup, private key encrypted with AES-256-GCM in the database, survives server restarts
- **Rotating key store** — support for multiple relayer keys for high-throughput scenarios
- **Profile integration** — near-kit profile lookup, NEAR Social fallback
- **near.email** — `.near` accounts automatically get `localpart@near.email`; non-`.near` accounts created without email
- **Contract view endpoint** — server-side read-only contract calls for authenticated users

```
┌─ Web App ─────────────────────────────────────────┐
│  siwnClient({ recipient, networkId, apiKey })      │

│  authClient.near.relayTransaction(...)             │
└──────────────────┬────────────────────────────────┘
                   │ delegate action payload (base64)
                   ▼
┌─ Better Auth Server ──────────────────────────────┐
│  /near/relay — validates session + whitelist,      │
│                 wraps in tx, signs with relayer     │
│                 key, broadcasts via near-kit        │
│  /near/relay-status/:txHash                        │
│  /near/relayer-info                                │
│  /near/nonce, /near/verify, /near/profile, etc.   │
│  /near/view — server-side contract reads           │
└──────────────────┬─────────────────────────────────┘
                   │ broadcast_tx_commit
                   ▼
              NEAR Protocol (on-chain)
```

## Plugin Options

```typescript
siwn({
  recipient: string;                    // NEP-413 recipient (required)
  requireFullAccessKey?: boolean;       // default: false
  getNonce?: () => Promise<Uint8Array>;
  validateNonce?: (nonce: Uint8Array) => boolean;
  validateRecipient?: (recipient: string) => boolean;
  validateMessage?: (message: string) => boolean;
  getProfile?: (accountId: string) => Promise<Profile | null>;
  validateLimitedAccessKey?: (args: { accountId, publicKey, recipient? }) => Promise<boolean>;

  // API key for RPC (near-kit compatible providers)
  apiKey?: string;                      // or process.env.FASTNEAR_API_KEY

  // Relayer
  relayer?: {
    accountId?: string;                  // explicit mode (production)
    privateKey?: string;                 // single key (base64 ed25519:...)
    privateKeys?: string[];              // multiple keys for RotatingKeyStore
    whitelistedContracts?: string[];
    maxGasPerTransaction?: string;
    maxDepositPerTransaction?: string;
  };
  // When accountId + privateKey omitted → ephemeral mode
})
```

## Email Handling

- `.near` accounts: `efiz.near` → `efiz@near.email` (automatic)
- Non-`.near` accounts: no email set; users add email via other Better Auth plugins
- The `anonymous` and `emailDomainName` options have been removed

## Relayer Modes

### Ephemeral (default)

When `relayer` is provided without `accountId`/`privateKey`:

1. On startup, checks DB for existing `relayerKey` record
2. If found: decrypts private key using `BETTER_AUTH_SECRET` as KEK (HKDF-SHA256 → AES-256-GCM), recovers same accountId
3. If not found: generates ED25519 keypair via `near-kit`'s `generateKey()`, derives implicit accountId (hex of public key), encrypts + stores in DB, logs accountId
4. Private key lives only in process memory — never in env vars or config files

Startup log:
```
[siwn] Relayer created in EPHEMERAL mode: 7a3c4b5c... (mainnet)
[siwn] Fund this account with NEAR to enable gasless relay
[siwn] Private key is encrypted in DB — persists across restarts
```

### Explicit (production)

When `accountId` + `privateKey`/`privateKeys` are provided in config — for named, persistent relayer accounts like `relayer.myapp.near`.

### Rotating Keys

When `privateKeys` array is provided with multiple keys, a `RotatingKeyStore` is used to cycle through keys round-robin, eliminating nonce collisions for high-throughput relayers.

## NEP-413 Verify Flow

The verify endpoint accepts a structured request (not a single `authToken` string):

```typescript
POST /near/verify
{
  signedMessage: { accountId, publicKey, signature },
  message: "Sign in to example.near",
  recipient: "example.near",
  nonce: "<hex-encoded 32 bytes>",
  accountId: "alice.near"
}
```

Server-side verification uses `verifyNep413Signature()` from near-kit with:
- Built-in 5-minute nonce expiration (from embedded timestamp in nonce)
- Additional DB-based nonce replay detection as defense-in-depth
- Access key validation (full access or function call key matching recipient)

## Delegate Action Flow (NEP-366)

```
1. User signs in via SIWN → gets function-call access key (FAK)
2. FAK is scoped to the recipient contract
3. Client calls buildSignedDelegateAction({ receiverId, actions })
   - Uses near-kit TransactionBuilder .delegate() method
   - Returns base64 payload string
4. Client calls relayTransaction({ payload })
   - Server decodes payload via decodeSignedDelegateAction()
   - Validates: session authenticated, sender matches, receiver whitelisted
   - Wraps delegate action via near-kit .signedDelegateAction().send()
   - Transaction executes on-chain as if the user sent it
   - Relayer pays gas
   - predecessor_id = user's accountId
```

## Profile Lookup

Profile lookup tries FastNear KV first, falls back to NEAR Social:

1. `GET https://kv.main.fastnear.com/v0/latest/social.near/{accountId}/profile/**`
2. Fallback: `POST https://api.near.social/get` with `{ keys: ["{accountId}/profile/**"] }`

## Encryption Details

- **KEK derivation**: HKDF-SHA256 with `BETTER_AUTH_SECRET`, salt `better-near-auth-relayer`, info empty
- **Encryption**: AES-256-GCM with 12-byte random IV
- **Storage**: `encryptedPrivateKey` (base64 of ciphertext) + `iv` (base64 of IV) in `relayerKey` table
- **Trust model**: Same as Better Auth session tokens — DB access + `BETTER_AUTH_SECRET` = full access

## Wallet Connection (Client)

Uses near-kit's `fromNearConnect()` adapter with `@hot-labs/near-connect`:

- `connector.connect()` — opens wallet picker
- `connector.disconnect()` — disconnects
- `near.signMessage({ message, recipient, nonce })` — NEP-413 signing
- `near.transaction(accountId).functionCall(...).delegate()` — delegate action signing
- `connector.getConnectedWallet()` — current account
- `connector.on("wallet:signIn", cb)` / `connector.on("wallet:signOut", cb)` — event listeners

## RPC Calls

All on-chain queries and transaction broadcasting use near-kit's `Near` instance:

- Configured with `{ network, headers: apiKey ? { Authorization: "Bearer ..." } : undefined }`
- `near.accountExists()`, `near.getAccessKey()`, `near.getAccount()`, `near.view()`, `near.getTransactionStatus()`
- Transaction broadcast handled internally by near-kit's `.send()` method
